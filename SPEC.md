# Клиника ПЛЮС — Спецификация проекта

> Единый источник правды для разработки. Положите этот файл в корень репозитория как `SPEC.md`. Claude Code должен читать его перед любыми изменениями и строго следовать архитектуре и бизнес-правилам.

---

## 1. О продукте

Корпоративное приложение научно-образовательного подразделения детской клиники «ПЛЮС» для врачей.

**Пользователи:** до 100 врачей на старте, до 500 при монетизации.
**Роли:** `doctor` (врач) и `admin` (руководитель подразделения — наполняет контент).

**Пять разделов приложения врача:**
1. **Главная** — дашборд: ближайшее мероприятие, прогресс плана, новости, новый случай.
2. **Активности** — анонсы и календарь мероприятий, push-напоминания.
3. **Мой путь** — хаб с тремя вкладками: *План года · Дневник успеха · Архив*.
4. **Новости** — лента медицинских новостей за 1–2 недели.
5. **Случай из практики** — клиническая задача → тест → результат → разбор/статья.

## 5. Модель данных

См. рабочую схему в артефакте `src/db/schema.ts`. Ключевой принцип архитектуры:

### Разделение «план года» и «постоянный профиль»

```
ПОСТОЯННЫЙ ПРОФИЛЬ (живёт все годы работы врача)
  ├── successWins     — дневник успеха (append-only, НЕ стирается)
  ├── balanceWheels   — колёса баланса по годам (start + end)
  └── reflections     — рефлексии (квартальные + годовые) по годам

ПЛАН ГОДА (обновляется ежегодно)
  └── plans (PK: userId + year)
      При загрузке нового года старый НЕ удаляется → становится архивом (read-only)
```

**Таблицы:**
- `users` (id, email, passwordHash, name, role, specialty)
- `plans` — **PK составной: (userId, year)** — чтобы хранить планы за все годы
- `successWins` (id, userId, text, category, createdAt) — дневник, не стирается
- `balanceWheels` (id, userId, year, phase: 'start'|'end', values jsonb)
- `reflections` (id, userId, year, scope: 'Q1'|'Q2'|'Q3'|'Q4'|'year', text)
- `activities`, `news`, `cases`, `caseSubmissions`, `deviceTokens`

> ⚠️ Текущий `schema.ts` использует `plans.userId` как одиночный PK. **Изменить на составной (userId, year)** и добавить таблицы `successWins`, `balanceWheels`, `reflections`.

## 6. Ключевые бизнес-правила

### 6.1. Архив прошлых лет — позитивная подача
- Если за год выполнено **≥ 60%** → показываем процент + список достижений.
- Если **< 60%** → процент **НЕ показываем**. Вместо него: «За {год} вы:» + список того, что реально сделано, и тёплая подпись. Никакого негатива и упоминания невыполненного.

### 6.2. Push-уведомления о мероприятиях
Для каждого события — расписание из нескольких напоминаний:
- **за 2 часа**, **за 5 минут**, **в момент начала**.
- Работают при закрытом приложении (системные push через Expo Push API).
- У врача есть настройка, какие напоминания получать.
- Серверный планировщик (cron в `apps/api`) рассылает push по `deviceTokens`.

### 6.3. Безопасность (обязательно)
- Все тела/параметры запросов валидируются Zod.
- Admin-роуты защищены `requireRole("admin")`.
- План доступен только владельцу или админу — **проверка IDOR** (`sub === userId || role === admin`).
- Пароли bcrypt (cost 12). JWT exp = 7 дней.
- Rate limit на `/auth/*` (10 req/мин на IP).
- HTML-поля экранируются (`he.escape`) перед сохранением (анти-XSS).
- Только Drizzle ORM, без сырого SQL.
- Self-registration только как `doctor`; `admin` создаётся вручную.

### 6.4. Клинический случай
- Одна попытка на врача на кейс (идемпотентность, `caseSubmissions` уникален по userId+caseId).
- После отправки: счёт, разбор по каждому вопросу, ссылка/текст статьи.

## 7. API (основные эндпоинты)

```
POST   /auth/register            (doctor only)
POST   /auth/login
GET    /plans/me                 (свой план текущего года)
GET    /plans/:userId            (admin или сам)
PUT    /plans/:userId            (admin загружает план)
PATCH  /plans/me/tasks/:taskId   (статус задачи)
GET    /plans/me/archive         (все годы врача)
POST   /wins                     (добавить успех)
GET    /wins                     (дневник успеха)
PUT    /balance/:year/:phase     (колесо баланса)
PUT    /reflections/:year/:scope (рефлексия)
GET    /activities               (?from=&to=)
POST   /activities               (admin)
GET    /news                     (?page=&limit=)
POST   /news                     (admin)
GET    /cases
POST   /cases/:id/submit
POST   /devices/register         (push-токен)
```

## 9. Порядок разработки (параллельные треки)

**Трек A — Backend + БД (фундамент для всего):**
1. Поднять Supabase/Neon, прописать `.env` (JWT_SECRET, DATABASE_URL).
2. Обновить `schema.ts` (составной PK плана + новые таблицы), сделать миграции Drizzle.
3. Реализовать роуты так, чтобы прошли тесты (`bun test`).

**Трек B — Mobile (приложение врача):**
1. Инициализировать Expo + expo-router, настроить тему (цвета из §4).
2. Экраны по прототипу: 5 табов + потоки (план, кейс, дневник, архив).
3. Подключить API-клиент и auth.
4. Настроить expo-notifications + регистрацию push-токена.

**Трек C — Admin (после базового API):**
1. React + Vite, авторизация админа.
2. Управление планами, мероприятиями, новостями, кейсами.

**Финал:** EAS Build → публикация в App Store ($99/год) и Google Play ($25).
